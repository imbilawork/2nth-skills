TWENTY CRM — DEPLOYMENT PATTERN

TWENTY CRM
RAILWAY + CLOUDFLARE

Self-host Twenty CRM on Railway with Cloudflare R2 for file storage, Cloudflare DNS for your subdomain, and Zero Trust Access to gate it to your team only. Validated env vars, step-by-step deployment, zero VPS management.

Twenty CRM Railway Cloudflare R2 CF DNS + Zero Trust ~$15–25/month Validated pattern
01 — RATIONALE

WHY THIS STACK

Twenty CRM requires PostgreSQL, Redis, and a persistent process — none of which exist natively on Cloudflare. The closest alternative (CF Containers) is beta, has ephemeral disk, and would still need external Postgres and Redis. For a production CRM you rely on daily, that's the wrong tradeoff.

Railway is the clean answer: Docker-native, managed Postgres and Redis, persistent volumes, one-command deploys from the Twenty Docker image, and a custom domain in two minutes. Cloudflare wraps the outside — DNS, TLS, and Zero Trust access control so only your team can reach it.

LayerPlatformWhy
Application (server + worker)RailwayDocker-native, managed deps, one-command deploy
PostgreSQL databaseRailway PostgresManaged, auto-backups, same network as app
Redis (queues + cache)Railway Redis or UpstashBull queue support, no external broker
File storageCloudflare R2S3-compatible, zero egress cost, global CDN

Access controlCloudflare Zero TrustTeam-only gate, no Twenty auth complexity needed

twenty-server is the main NestJS application. It serves both the API (/api, /graphql) and the compiled React frontend as static files from the same port 3000. There is no separate frontend deployment needed — Twenty bundles everything into one Docker image.

twenty-worker runs from the same Docker image but with a different start command (yarn worker:prod). It processes background jobs — email syncing, webhook delivery, reminder tasks — from the Bull queue backed by Redis.

04 — CLOUDFLARE R2

R2 BUCKET SETUP

Twenty stores file attachments, profile pictures, and workspace logos in S3-compatible object storage. R2 is zero-egress-cost and already in your CF account.

Create the bucket

# Via Wrangler (run from 2nth-site project)
npx wrangler r2 bucket create twenty-crm-files

# Verify
npx wrangler r2 bucket list

Create an R2 API token

In the Cloudflare dashboard: R2 → Manage R2 API Tokens → Create API Token

  • Permissions: Object Read & Write
  • Bucket: restrict to twenty-crm-files only
  • Save the Access Key ID and Secret Access Key — shown once

Get your R2 endpoint

# Your R2 endpoint format:
https://<ACCOUNT_ID>.r2.cloudflarestorage.com

# Find your Account ID:
# Cloudflare dashboard → right sidebar → Account ID
# Or: npx wrangler whoami
CORS for file uploads

Twenty uploads files directly from the browser to R2. You need to set a CORS policy on the bucket allowing https://20crm.client.2nth.ai as an origin. Do this in CF dashboard → R2 → your bucket → Settings → CORS policy.

// R2 CORS policy (paste in CF dashboard)
[
  {
    "AllowedOrigins": ["https://20crm.client.2nth.ai"],
    "AllowedMethods": ["GET", "PUT", "POST", "DELETE", "HEAD"],
    "AllowedHeaders": ["*"],
    "MaxAgeSeconds": 3600
  }
]
05 — RAILWAY DEPLOYMENT

RAILWAY DEPLOYMENT

Two services from the official Twenty Docker image (twentycrm/twenty:latest) — one for the server, one for the worker — plus managed Postgres and Redis.

Option A — Railway template (fastest)

Railway maintains an official Twenty CRM template. Visit railway.app/template/twenty-crm and click Deploy. It scaffolds all 4 services. You still need to set the env vars below before the first deploy succeeds.

Option B — Manual setup via CLI

1
Create a new Railway project
railway init twenty-crm
cd twenty-crm
2
Add PostgreSQL and Redis
railway add --service postgres
railway add --service redis
3
Add twenty-server service
In Railway dashboard → New Service → Docker Image → twentycrm/twenty:latest
Set start command: yarn start:prod
Set pre-deploy command: yarn database:init:prod
Expose port: 3000
4
Add twenty-worker service
Same Docker image: twentycrm/twenty:latest
Set start command: yarn worker:prod
No port needed — this service processes queues, not HTTP requests.
5
Set environment variables
Add all vars from the reference below to both twenty-server and twenty-worker. See Section 06.
6
Add custom domain to twenty-server
Railway → twenty-server → Settings → Custom Domain → 20crm.client.2nth.ai
Railway gives you a CNAME target. You'll point Cloudflare DNS at it in Section 07.
7
Deploy
railway up

# Watch logs
railway logs --service twenty-server
Pre-deploy command is critical

The yarn database:init:prod pre-deploy command runs database migrations before the server starts. Without it, the server will crash on first boot with migration errors. Set this on the twenty-server service only — not the worker.

06 — ENV VARS REFERENCE

ENVIRONMENT VARIABLES

Set these on both twenty-server and twenty-worker in Railway. Railway injects Postgres and Redis connection strings automatically via reference variables — use the ${{Postgres.DATABASE_URL}} syntax to avoid hardcoding credentials.

Infrastructure vars — cannot be changed via UI

These 5 vars are read at process start before the database is available. They must be set as Railway env vars — they cannot be set through Twenty's admin panel even when IS_CONFIG_VARIABLES_IN_DB_ENABLED=true.

REQUIRED — Core infrastructure
NODE_ENV=production

# Railway injects these automatically — use reference syntax
PG_DATABASE_URL=${{Postgres.DATABASE_URL}}
REDIS_URL=${{Redis.REDIS_URL}}

# Generate: openssl rand -base64 32
APP_SECRET=replace_with_64_char_random_string

# Must match your custom domain exactly — set BEFORE first deploy
SERVER_URL=https://20crm.client.2nth.ai
FRONTEND_URL=https://20crm.client.2nth.ai
REQUIRED — Cloudflare R2 file storage
# Use S_3 (not s3 or S3) — Twenty's enum value
STORAGE_TYPE=S_3
STORAGE_S3_REGION=auto
STORAGE_S3_NAME=twenty-crm-files
STORAGE_S3_ENDPOINT=https://<ACCOUNT_ID>.r2.cloudflarestorage.com
STORAGE_S3_ACCESS_KEY_ID=<your-r2-access-key-id>
STORAGE_S3_SECRET_ACCESS_KEY=<your-r2-secret-access-key>
REQUIRED — Email via Resend SMTP
EMAIL_DRIVER=SMTP
EMAIL_FROM_ADDRESS=hello@2nth.ai
EMAIL_FROM_NAME=2NTH CRM
EMAIL_SMTP_HOST=smtp.resend.com
EMAIL_SMTP_PORT=465
EMAIL_SMTP_USER=resend
EMAIL_SMTP_PASSWORD=<your-resend-api-key>
IS_EMAIL_VERIFICATION_REQUIRED=false
RECOMMENDED — Security & privacy
# Enable password auth (used alongside CF Access)
AUTH_PASSWORD_ENABLED=true

# Single workspace — we're not a SaaS
IS_MULTIWORKSPACE_ENABLED=false
IS_WORKSPACE_CREATION_LIMITED_TO_SERVER_ADMINS=true
IS_BILLING_ENABLED=false

# Disable telemetry sending to Twenty's servers
ANALYTICS_ENABLED=false
TELEMETRY_ENABLED=false

# Disable sign-in prefill (security)
SIGN_IN_PREFILLED=false
OPTIONAL — AI providers for Twenty's native AI features
# Twenty has native AI features — wire to Anthropic for best results
ANTHROPIC_API_KEY=<your-anthropic-key>

# Or use a custom gateway (e.g. CF AI Gateway proxying Anthropic)
# AI_PROVIDERS='{"cf-gateway":{"type":"openai-compatible","baseUrl":"https://gateway.ai.cloudflare.com/v1/...","apiKey":"..."}}'
OPTIONAL — Token expiry (adjust to taste)
ACCESS_TOKEN_EXPIRES_IN=30m
LOGIN_TOKEN_EXPIRES_IN=15m
REFRESH_TOKEN_EXPIRES_IN=90d
FILE_TOKEN_EXPIRES_IN=1d

07 — CLOUDFLARE DNS

CLOUDFLARE DNS

Point 20crm.client.2nth.ai at your Railway deployment. Railway provides a CNAME target from the custom domain settings of your twenty-server service.

# In Cloudflare DNS dashboard → 2nth.ai zone
# Add a CNAME record:

Type:    CNAME
Name:    20crm.client     (creates 20crm.client.2nth.ai)
Target:  <your-railway-cname>.up.railway.app
Proxy:   ✓ Proxied (orange cloud)
TTL:     Auto
Proxied is required for Zero Trust

Cloudflare Access (Zero Trust) only intercepts requests that pass through CF's proxy. If you set the record to DNS-only (grey cloud), the access gate won't fire and the CRM will be publicly accessible. Always proxy this record.

After adding the CNAME, go back to Railway → twenty-server → custom domain and verify the domain is confirmed. Railway validates via a TXT record that Cloudflare resolves automatically.

08 — CLOUDFLARE ZERO TRUST

ZERO TRUST ACCESS

Lock the CRM to your team only. Every browser request to 20crm.client.2nth.ai passes through CF Access before hitting Railway. No valid CF Access session = redirect to a login page — Twenty's own auth never even sees the request.

Create the Access application

# Cloudflare Zero Trust dashboard:
# Access → Applications → Add an Application → Self-hosted

Application name:  2NTH CRM
Application URL:   https://20crm.client.2nth.ai
Session duration:  24 hours (adjust to taste)

# Under Identity Providers:
# Add "One-time PIN" (email OTP) — works immediately, no OAuth app needed
# Optional: add Google Workspace for @2nth.ai SSO

Create an access policy

# Policy name: 2nth Team
# Action: Allow
# Rules:

# Option A — allow specific emails:
Include → Emails → craig@2nth.ai, craig@b2bs.co.za, [team emails]

# Option B — allow entire domain (cleaner for growing teams):
Include → Email domain → 2nth.ai

# Option C — combine both domains:
Include → Emails → [list]
OR
Include → Email domain → b2bs.co.za
No VPN, no passwords to share

Team members visit the URL, enter their email, and get a one-time code. CF Access validates the code, sets a JWT cookie, and forwards the request to Railway. Twenty sees an already-authenticated user. You can also disable Twenty's own auth entirely and rely purely on CF Access as the gate.

Bypass for API/webhook access

If you want your CF Workers or agents to call Twenty's GraphQL API programmatically, they need to bypass CF Access. Use a service token:

# Zero Trust → Access → Service Auth → Service Tokens → Create
# Store the Client ID and Client Secret as Worker secrets

# Use in Worker requests to Twenty:
fetch("https://20crm.client.2nth.ai/api/graphql", {
  headers: {
    "CF-Access-Client-Id": env.TWENTY_CF_CLIENT_ID,
    "CF-Access-Client-Secret": env.TWENTY_CF_CLIENT_SECRET,
    "Authorization": `Bearer ${env.TWENTY_API_TOKEN}`,
    "Content-Type": "application/json"
  },
  // ...
})
09 — EMAIL CONFIGURATION

EMAIL CONFIGURATION

Twenty sends transactional email for magic links, workspace invitations, and task reminders. Use Resend — already in the 2nth.ai stack, same domain, same DKIM/SPF.

# Railway env vars:
EMAIL_DRIVER=SMTP
EMAIL_FROM_ADDRESS=hello@2nth.ai
EMAIL_FROM_NAME=2NTH CRM
EMAIL_SMTP_HOST=smtp.resend.com
EMAIL_SMTP_PORT=465
EMAIL_SMTP_USER=resend
EMAIL_SMTP_PASSWORD=<resend-api-key>   # re:prefix keys work too
IS_EMAIL_VERIFICATION_REQUIRED=false
Resend API key scope

Create a separate Resend API key scoped to sending only — don't reuse the 2nth.ai platform key. Name it twenty-crm in the Resend dashboard so it's auditable separately.

Optional — Gmail / Microsoft integration

Twenty can sync your Gmail and Google Calendar into the CRM. This requires a Google Cloud OAuth app with the Gmail and Calendar APIs enabled. Set these vars after the initial deployment is stable:

AUTH_GOOGLE_ENABLED=true
AUTH_GOOGLE_CLIENT_ID=<oauth-client-id>
AUTH_GOOGLE_CLIENT_SECRET=<oauth-client-secret>
AUTH_GOOGLE_CALLBACK_URL=https://20crm.client.2nth.ai/auth/google/redirect
AUTH_GOOGLE_APIS_CALLBACK_URL=https://20crm.client.2nth.ai/auth/google-apis/get-access-token
MESSAGING_PROVIDER_GMAIL_ENABLED=true
CALENDAR_PROVIDER_GOOGLE_ENABLED=true

11 — UPDATES & BACKUPS

UPDATES & BACKUPS

Updating Twenty

# Railway dashboard → twenty-server → Deployments → Redeploy
# OR update the image tag in service settings from 'latest' to a specific version

# Pin to a specific version (recommended for production):
# Image: twentycrm/twenty:0.43.0   (check releases at github.com/twentyhq/twenty)

# Railway will:
# 1. Pull the new image
# 2. Run yarn database:init:prod (migrations)
# 3. Perform a rolling deploy with zero downtime
Always check the Twenty changelog before updating

Twenty moves fast. Some minor versions include breaking database migrations. Read the release notes at github.com/twentyhq/twenty/releases before updating. Take a manual Postgres snapshot from Railway before any version jump.

Database backups

# Railway Postgres has built-in point-in-time recovery on Hobby+ plans
# For manual snapshots before a major update:

railway connect postgres   # Opens a psql shell

# Or via pg_dump:
pg_dump $DATABASE_URL > twenty_backup_$(date +%Y%m%d).sql

# Store snapshot in R2:
npx wrangler r2 object put twenty-backups/twenty_backup_$(date +%Y%m%d).sql \
  --file=twenty_backup_$(date +%Y%m%d).sql

Monitoring

# Watch server logs
railway logs --service twenty-server

# Watch worker logs (queue processing)
railway logs --service twenty-worker

# Health endpoint
curl https://20crm.client.2nth.ai/healthz

13 — TROUBLESHOOTING

TROUBLESHOOTING

SymptomLikely causeFix
Server crashes on startupFree plan RAM limit (512 MB)Upgrade Railway to Hobby plan ($5/month)
Migration error on first bootPre-deploy command not setSet yarn database:init:prod as pre-deploy on twenty-server
Files not uploadingWrong STORAGE_TYPE valueUse S_3 exactly — not s3 or S3
Files upload but broken URLsR2 CORS not configuredAdd CORS policy on R2 bucket allowing your domain
Emails not sendingWrong SMTP port or authResend uses port 465 with SSL; user is literally resend
CF Access loop / infinite redirectDomain not proxied via CFEnsure DNS record is proxied (orange cloud), not DNS-only
Workers can't call Twenty APICF Access blocking programmatic requestsUse CF Access service token in request headers
SERVER_URL mismatch errorsURL set after first deploySERVER_URL must be set before first migration run — nuke DB and redeploy
GraphQL endpoint 401API token expired or wrongRegenerate token in Twenty → Settings → API
Worker not processing jobsREDIS_URL not set on worker serviceEnsure all env vars are set on both services, not just server
Nuclear reset

If the database ends up in a broken state (bad migration, wrong SERVER_URL baked in): drop and recreate the Railway Postgres database, clear the Redis instance, and redeploy. Twenty's migrations run fresh. You'll need to redo workspace setup but all env vars stay the same.
