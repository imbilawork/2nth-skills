Architecture Decision Framework

Before selecting any tool, answer these five questions:

  1. Data Volume & Velocity — How much data, how fast does it arrive?
  2. Query Patterns — Ad-hoc exploration vs. fixed dashboards vs. embedded analytics?
  3. Team Capability — SQL-literate analysts? Dedicated data engineers? Self-service business users?
  4. Latency Requirements — Real-time, near-real-time, or batch is fine?
  5. Budget Constraints — Cloud-managed vs. self-hosted? Per-seat licensing tolerance?

DIMENSIONAL MODELING

Dimensional modeling is the design discipline that makes data queryable by business users. Fact tables hold measurements. Dimension tables hold descriptive context. The grain is everything.

ConceptRule
GrainDefine the grain FIRST — one row = one what?
Fact tablesContain measures (numeric, additive). Prefix with fct_
Dimension tablesContain descriptive attributes. Prefix with dim_
Surrogate keysAlways use integer surrogates, never natural keys as PKs
Date dimensionAlways build a dedicated dim_date with fiscal calendar support
Junk dimensionsCombine low-cardinality flags into a single junk dimension
Degenerate dimensionsOrder/invoice numbers live in the fact table itself

SCD Type Selection

SCD TypeHow It WorksUse When
Type 1Overwrite old valueHistory not needed (e.g. phone number corrections)
Type 2New row + effective datesMust preserve history (e.g. customer region changes)
Type 3Add "previous value" columnOnly need one prior value (rarely used)

ETL/ELT & DBT

ELT (Extract, Load, Transform) is the default choice for modern cloud warehouses. Load raw data first, then transform in-warehouse using dbt. This approach is more flexible, cheaper, and easier to debug than traditional ETL.

The recommendation

For new builds: Airbyte for ingestion, dbt Core for transformation, Dagster for orchestration. All open-source, all production-proven.

dbt Project Structure

models/
  staging/          ← one model per source table, minimal transformation
    stg_orders.sql
    stg_customers.sql
  intermediate/     ← joins and business logic, not exposed to BI
    int_order_items_enriched.sql
  marts/            ← final fact and dimension tables, BI-ready
    fct_orders.sql
    dim_customers.sql
    dim_date.sql

Essential dbt Tests

# models/marts/schema.yml
models:
  - name: fct_orders
    columns:
      - name: order_id
        tests:
          - not_null
          - unique
      - name: customer_id
        tests:
          - not_null
          - relationships:
              to: ref('dim_customers')
              field: customer_id
      - name: order_status
        tests:
          - accepted_values:
              values: ['pending', 'complete', 'cancelled', 'returned']

Choose the warehouse that fits your volume, budget, and team. Open-source options cover most SME requirements at zero licensing cost.

SA/emerging market recommendation

ClickHouse is the top pick for high-volume analytics on self-hosted infrastructure. It's orders of magnitude faster than PostgreSQL for analytical queries and costs nothing per seat. PostgreSQL is fine for <100GB and teams already running it for their application database.

DATA GOVERNANCE

Governance is not a compliance checkbox. It's the operational discipline that keeps data trustworthy as the organisation grows.

Minimum Viable Governance

  • Data Catalog — Document every table, column, and metric definition
  • Ownership — Every dataset has a named owner accountable for quality
  • Quality Tests — Automated checks on freshness, completeness, and schema
  • Access Control — Role-based access, principle of least privilege
  • Lineage — Track data flow from source to dashboard (dbt provides this)
  • Retention Policy — Define how long raw/processed data is kept

POPIA Compliance (SA Context)

  • Personal data must be processed lawfully; data minimisation applies
  • Store consent records alongside PII with timestamps
  • Design soft-delete patterns to support deletion requests
  • Document and justify any data leaving SA borders
